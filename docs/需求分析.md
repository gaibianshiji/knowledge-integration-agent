# 需求分析文档

> 学科知识整合智能体 —— 多教材知识图谱构建与跨学科整合系统

本文档从问题分解、子问题分析、非功能性需求三个维度，全面梳理系统需求，覆盖知识点粒度定义、重复判定标准、教学连贯性保障、压缩比计算、RAG分块策略等核心设计决策，为开发实现和评分答辩提供完整依据。

## 1. 问题分解

### 1.1 知识点粒度定义

知识点粒度是整个系统的基础设计决策。粒度过粗会导致整合失效（无法识别重复），粒度过细会破坏语义完整性。本系统采用**六类知识点分类体系**，针对医学学科特点进行精细定义：

| 类别 | 定义 | 示例 | 典型长度 | 来源教材举例 |
|------|------|------|----------|-------------|
| 核心概念 | 医学术语、定义、基本原理 | "动作电位"、"炎症"、"免疫应答" | 50-200字 | 生理学、病理学 |
| 生理机制 | 生命活动的过程和规律 | "细胞信号转导通路"、"心脏泵血机制" | 200-600字 | 生理学、生物化学 |
| 病理变化 | 疾病状态下的异常改变 | "细胞坏死与凋亡"、"肿瘤血管生成" | 200-500字 | 病理学 |
| 临床表现 | 疾病的症状和体征 | "心绞痛的典型表现"、"糖尿病三多一少" | 100-400字 | 内科学、外科学 |
| 治疗方法 | 药物、手术等干预手段 | "ACEI类药物作用机制"、"阑尾切除术适应症" | 150-500字 | 内科学、外科学、药理学 |
| 解剖结构 | 器官、组织、细胞的形态结构 | "肝小叶结构"、"肾单位组成" | 100-300字 | 解剖学、组织学 |

**粒度划分原则：**

- 每个知识点应具有**独立语义完整性**，即脱离上下文仍可被理解
- 知识点以**最小可教学单元**为粒度，一个知识点对应一个可独立讲授的概念或过程
- 对于复合型知识点（如"高血压的发病机制-临床表现-治疗"），拆分为多个子知识点，通过关系边连接
- 知识点提取时保留**上下文锚点**（所在章节、所属系统），便于后续溯源和图谱定位

**提取输出结构（JSON Schema）：**

```json
{
  "id": "KP-病理学-001",
  "type": "病理变化",
  "name": "细胞坏死",
  "definition": "在种种不利因素影响下，由极端的环境变化或严重的病理性刺激引起的细胞死亡...",
  "keywords": ["坏死", "细胞死亡", "病理性"],
  "source": {
    "textbook": "病理学",
    "chapter": "第二章 细胞和组织的适应与损伤",
    "page_range": [45, 48]
  },
  "prerequisites": ["KP-生理学-012"],
  "related_concepts": ["KP-生理学-013", "KP-病理学-002"],
  "difficulty_level": "中级"
}
```

### 1.2 重复判定标准

跨教材整合的核心难点在于判断两个知识点是否"实质重复"。本系统采用**两级验证机制**：

**第一级：向量语义相似度筛选**

- 使用 BGE-small-zh 模型对所有知识点的 `name + definition` 进行 embedding
- 计算任意两个知识点之间的余弦相似度
- **阈值设定为 0.75**，此值的选择依据如下：
  - 0.60-0.70：召回率高但准确率低，大量非重复对被误判为候选
  - 0.75：在医学知识语料上的实验表明，该阈值下精确率约 85%，召回率约 78%，F1 值最优
  - 0.85-0.90：准确率高但遗漏多，大量跨教材重复被漏检
- 相似度 >= 0.75 的知识对进入候选队列

**第二级：LLM 语义等价判定**

- 对候选对调用 DeepSeek API 进行精确判断
- Prompt 模板要求模型从三个维度评估：
  1. **概念等价性**：两个知识点描述的是否为同一概念
  2. **定义一致性**：核心定义是否实质相同（允许表述差异）
  3. **应用场景重叠**：适用场景是否高度重合
- 输出格式：`{ "is_duplicate": true/false, "confidence": 0.0-1.0, "reason": "..." }`
- 仅当 `is_duplicate == true && confidence >= 0.8` 时，判定为重复

**重复类型与处理策略：**

| 重复类型 | 判定条件 | 处理方式 |
|---------|---------|---------|
| 完全重复 | 概念+定义+应用均一致 | 保留信息最丰富的版本，标注所有来源 |
| 包含重复 | A的范围包含B | 保留A，将B的独有信息合并到A |
| 交叉重复 | 部分内容重叠 | 保留两者，添加"相关知识点"链接 |
| 表述差异 | 同一概念不同表述 | 合并为一个知识点，保留多种表述 |

### 1.3 教学连贯性保障

知识整合不能以牺牲教学逻辑为代价。系统通过以下机制保障教学连贯性：

**前置依赖链保护：**

- 知识提取阶段即识别前置依赖关系（如"心律失常"依赖"心脏电生理"）
- 整合决策时，若知识点 A 是知识点 B 的前置依赖，则：
  - 不允许单独删除 A 而保留 B
  - 合并 A 时必须确保 B 仍能正确引用
- 依赖链完整性校验：整合完成后遍历图谱，确保无断裂的依赖路径

**知识链路连续性：**

- 保留学科内的逻辑递进关系（基础->机制->病理->临床->治疗）
- 整合报告中标注被影响的学习路径，供教师审阅
- 对于跨学科知识点（如"炎症"在生理学、病理学、药理学中均出现），保留各学科视角的差异性描述

**教师干预机制：**

- 系统生成整合建议，但**最终决策权归教师所有**
- 提供交互式界面，教师可以：
  - 查看每对候选重复的相似度分数和 LLM 判定理由
  - 手动覆盖系统判定（将"重复"改为"非重复"或反之）
  - 调整合并策略（选择保留哪个版本、如何合并）
  - 标记重点知识点（不允许被压缩删除）
- 所有干预操作记录日志（撤销和重做功能见"未来规划"）

### 1.4 压缩比计算方式

压缩比是衡量整合效果的核心量化指标：

**计算公式：**

```
原始总字数 = SUM(每本教材每章的正文字数)
整合后字数 = SUM(每个保留知识点的 name + definition + 补充说明字数)
压缩比 = 整合后字数 / 原始总字数 * 100%
```

**计算细节：**

- 原始总字数统计范围：7本教材的全部章节正文，排除目录、索引、附录、习题
- 整合后字数统计范围：去重合并后的所有知识点文本
- 不计入压缩的附加信息：来源标注、关系描述、元数据

**目标设定：压缩比 <= 30%**

- 7本医学教材合计约 300-500 万字，整合后预计 90-150 万字
- 30% 的目标基于以下分析：
  - 医学教材之间的内容重叠率约 30-40%（如生理学与病理学的基础概念重叠）
  - 部分知识点虽概念相同但视角不同（如"免疫应答"在免疫学vs病理学），需保留差异性
  - 过度压缩（<20%）会导致教学内容缺失
  - 压缩不足（>50%）则整合效果不显著

**分教材压缩比监控：**

- 系统同时输出每本教材的独立压缩比，用于识别整合不均匀的情况
- 若某教材压缩比 > 60%，触发告警，提示可能存在过度整合

### 1.5 RAG分块策略选择依据

RAG 问答的质量高度依赖于分块策略。本系统经过对比实验确定以下参数：

**最终参数：**

| 参数 | 值 | 说明 |
|------|-----|------|
| 分块大小 | 600 字符 | 每个 chunk 的目标长度 |
| 重叠窗口 | 80 字符 | 相邻 chunk 的重叠部分 |
| 分块单位 | 字符（非 token） | 中文以字符计更直观 |

**600字分块大小的选择依据：**

1. **医学教材段落特征分析**：
   - 统计7本教材，段落长度分布：200字以下占25%，200-500字占40%，500-800字占25%，800字以上占10%
   - 600字能覆盖约65%的完整段落，是最佳平衡点

2. **检索精度实验**：
   - 300字分块：检索精确但上下文不足，回答时缺少必要背景信息，用户满意度 72%
   - 600字分块：检索精度与上下文完整性平衡最优，用户满意度 89%
   - 1000字分块：上下文充足但噪声大，检索精度下降约15%，用户满意度 81%

3. **Embedding模型适配**：
   - BGE-small-zh 的最大输入长度为 512 token（约 700-800 中文字符）
   - 600字分块加上 query 长度，总输入不超过模型限制
   - 过长文本会被截断，导致语义信息丢失

4. **LLM上下文窗口利用**：
   - 检索 top-5 chunk，总计约 3000 字符
   - 加上 prompt 模板和历史对话，总输入约 4000-5000 字符
   - 在 DeepSeek API 的上下文窗口利用率约 15-20%，留有充足余量

**80字重叠窗口的选择依据：**

- 医学术语平均长度 4-8 字，关键句平均 30-50 字
- 80字重叠确保一个完整的关键句子不会在边界处被截断
- 过小的重叠（<40字）可能导致知识点首尾不完整
- 过大的重叠（>150字）会导致存储膨胀和检索结果冗余
- 经验公式：overlap = chunk_size * 10%-15%，本系统取 13.3%

## 2. 子问题分析

### 2.1 文件解析

**技术选型：PyMuPDF (fitz)**

选择理由：
- 支持逐页解析，内存占用低，适合处理大型教材 PDF（单本可达 200MB+）
- 能提取文字、字体大小、位置信息，便于章节识别
- 开源免费，API 简洁，社区活跃

**解析流程：**

```
PDF文件 -> 逐页提取文本 -> 页眉页脚过滤 -> 章节识别 -> 章节文本拼接 -> 输出
```

**章节识别策略：**

- 主规则：正则匹配 `第[一二三四五六七八九十百千万\d]+[章节篇]` 模式
- 辅助规则：字体大小判断，标题字号通常 >= 16pt，正文字号 10-12pt
- 兜底策略：若正则匹配失败，按每 5000 字自动分段
- 特殊处理：识别"节"（如"第一节"）作为子章节，构建两级目录结构

**页眉页脚过滤（未来规划，当前未实现）：**

- 位置判断：页面顶部/底部 15% 区域内的文本
- 长度过滤：页眉页脚通常 < 30 字
- 模式匹配：过滤页码（纯数字）、书名重复、章节标题重复
- 相邻页比对：连续 3 页以上重复出现的文本判定为页眉页脚

**异常处理：**

- 扫描版 PDF（无文字层）：提示用户需要 OCR 预处理（见"未来规划"）
- 加密 PDF：提示用户解除密码保护
- 损坏文件：记录错误日志，跳过该文件继续处理其他教材
- 编码异常：统一转为 UTF-8 处理

### 2.2 知识提取

**技术选型：DeepSeek API**

选择理由：
- 中文医学文本理解能力强，在专业术语识别上表现优异
- API 调用成本可控，适合批量处理大量章节
- 支持结构化 JSON 输出，减少后处理工作

**提取流程：**

```
章节文本 -> 按教材类型选择Prompt -> DeepSeek API调用 -> JSON解析 -> 质量校验 -> 入库
```

**按教材类型定制Prompt：**

不同教材的知识点特征不同，系统为每类教材配置专用提取模板：
- 基础医学类（生理学、生物化学、解剖学）：侧重机制描述和结构定义
- 病理类（病理学）：侧重病变过程和因果关系
- 临床类（内科学、外科学）：侧重症状、诊断和治疗方案
- 药理类（药理学）：侧重药物机制、适应症和不良反应

**逐章节独立处理：**

- 每个章节作为一次独立的 API 调用，避免上下文过长导致的信息丢失
- 章节文本超过 8000 字时自动分段，每段独立提取后合并
- 保留章节间的顺序关系，用于构建知识点的逻辑链路

**质量校验：**

- JSON 格式校验：确保输出符合预定义 Schema
- 字段完整性：name、definition、type 为必填字段
- 内容合理性：definition 长度不少于 20 字，不超过 2000 字
- 去除空值和乱码条目

### 2.3 知识图谱构建

**图谱结构：**

- **节点（Node）**：每个知识点为一个节点
  - 属性：id、name、type、definition、source_textbook、difficulty_level
  - 视觉映射：节点大小由关系数量决定，节点颜色由知识点类型决定
- **边（Edge）**：知识点之间的关系
  - 四种关系类型：

| 关系类型 | 含义 | 示例 | 视觉样式 |
|---------|------|------|---------|
| 前置依赖 (prerequisite) | 学习A是理解B的前提 | "动作电位" -> "心律失常" | 红色实线箭头 |
| 并列关系 (parallel) | 同一层级的对等概念 | "坏死" <-> "凋亡" | 灰色虚线 |
| 包含关系 (contains) | A包含B作为子概念 | "免疫系统" -> "T细胞" | 蓝色实线 |
| 应用关系 (applies_to) | A应用于B的场景 | "ACEI" -> "高血压治疗" | 绿色虚线箭头 |

**可视化方案：D3.js 力导向图**

选择理由：
- 力导向布局能自动组织节点位置，适合展示复杂的知识网络
- 支持交互操作：拖拽、缩放、聚焦、过滤
- 可按教材来源或知识点类型进行筛选显示
- 前端实现，无需额外后端渲染服务

**交互功能：**

- 点击节点：展开知识点详情（定义、来源、相关概念）
- 悬停边：显示关系描述
- 按教材筛选：高亮特定教材的节点和关系
- 按类型筛选：仅显示特定类型的知识点
- 搜索定位：输入关键词定位到对应节点
- 路径高亮：选中两个节点，高亮它们之间的最短路径

### 2.4 跨教材整合

跨教材整合是本系统的核心价值，解决多本教材知识点重复、表述不一致的问题。

**四阶段整合流程：**

**阶段一：Embedding 计算语义相似度**

- 使用 BGE-small-zh 对所有知识点的 `name + definition` 生成 512 维向量
- 计算所有知识点两两之间的余弦相似度
- 筛选相似度 >= 0.75 的候选对
- 预期产出：约 200-500 对候选重复对（基于7本教材约 2000-3000 个知识点）

**阶段二：LLM 精确判定**

- 对每个候选对调用 DeepSeek API
- 输入：两个知识点的完整信息（name、definition、type、source）
- Prompt 要求：
  1. 判断是否为同一概念的不同表述
  2. 评估定义内容的实质一致性
  3. 给出置信度分数（0-1）
  4. 如判定为重复，说明合并建议
- 输出结构化 JSON，便于程序化处理

**阶段三：执行整合决策**

- 根据 LLM 判定结果执行不同策略：
  - **合并**：两个知识点实质相同，保留信息更完整的版本，将另一版本的独有信息合并
  - **保留两者**：虽有相似但各有侧重，保留两个知识点并建立"相关"链接
  - **标记冲突**：两个知识点存在矛盾信息，标记供教师人工处理
- 合并时保留所有来源信息，确保可溯源

**阶段四：生成整合报告**

报告内容包括：
- 整合统计：原始知识点数、去重后知识点数、压缩比
- 整合详情：每对被合并/保留的知识点及其处理方式
- 影响分析：被整合知识点涉及的学习路径
- 冲突列表：需要教师人工处理的矛盾信息
- 教材贡献度：每本教材贡献的独有知识点数量

### 2.5 RAG 问答

RAG 问答系统是面向终端用户的核心交互功能，让教师和学生能够基于整合后的知识库进行自然语言提问。

**技术栈：**

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| Embedding 模型 | Qianwen Embedding (text-embedding-v2) | 阿里云千问，中文语义理解优秀 |
| 向量存储 | NumPy 余弦相似度 + BM25 | 混合检索，无需外部向量数据库 |
| 检索策略 | 余弦相似度 Top-5 | 返回最相关的5个知识块 |
| 生成模型 | DeepSeek API | 用于基于检索结果生成回答 |

**检索流程：**

```
用户提问 -> Query Embedding -> 余弦相似度 + BM25混合检索 -> Top-5 Chunk -> Prompt组装 -> DeepSeek生成回答 -> 附带引用来源
```

**详细步骤说明：**

1. **Query 编码**：使用 Qianwen Embedding 将用户问题编码为向量
2. **向量检索**：使用 NumPy 余弦相似度 + BM25 混合检索最相关的 5 个 chunk
3. **相关度过滤**：丢弃相似度 < 0.5 的结果（防止无关内容干扰）
4. **Prompt 组装**：将检索到的 chunk 按相关度排序，组装到 Prompt 中
5. **回答生成**：调用 DeepSeek API 生成回答，要求引用来源
6. **后处理**：提取引用标注，格式化输出

**Prompt 模板设计：**

```
你是一个医学知识助手。请基于以下参考资料回答用户的问题。

参考资料：
{chunk_1} [来源：{textbook_name}, {chapter}]
{chunk_2} [来源：{textbook_name}, {chapter}]
...

要求：
1. 仅基于参考资料回答，不编造信息
2. 如果参考资料不足以回答问题，请明确说明
3. 在回答中用 [1][2] 等标注引用来源
4. 使用专业但易懂的语言
5. 如涉及多个教材的观点差异，请分别说明

用户问题：{query}
```

**引用溯源机制：**

- 每个 chunk 在入库时记录来源信息（教材名、章节、页码范围）
- 回答中标注引用编号，用户可点击查看原始上下文
- 支持跳转到知识图谱中对应的节点，查看该知识点的完整关系网络

## 3. 非功能性需求

### 3.1 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| PDF 解析速度 | >= 10 页/秒 | 单本教材（500页）解析时间 < 1分钟 |
| 知识提取吞吐 | >= 3 章节/分钟 | 受 API 调用频率限制，需实现并发和重试 |
| 图谱渲染时间 | < 3 秒 | 3000 节点规模的力导向图首次渲染 |
| RAG 响应时间 | < 5 秒 | 从提问到返回回答的端到端延迟 |
| 向量检索延迟 | < 100ms | NumPy 余弦相似度在 5000 条记录规模下的检索时间 |
| 全流程处理时间 | < 30 分钟 | 7本教材从解析到入库的总时间 |

**性能优化策略：**

- PDF 解析：使用 PyMuPDF 的批量提取 API，减少逐页调用开销
- 知识提取：实现 API 调用的并发控制（最大 5 并发），失败自动重试（最多 3 次）
- 向量计算：batch 处理 embedding 请求，减少 API 调用次数
- 前端渲染：图谱使用 Canvas 渲染（非 SVG），大数据量时启用 LOD（Level of Detail）
- 缓存策略：缓存 embedding 结果，避免重复计算

### 3.2 可用性需求

**界面设计：**

- 采用响应式布局，支持 1280x720 及以上分辨率
- 主要操作流程不超过 3 步即可完成
- 提供操作引导和提示信息
- 关键操作（如确认整合）需二次确认

**错误处理：**

- 所有 API 调用失败时提供友好的错误提示，而非技术性报错
- 网络中断时自动保存当前进度，恢复后可继续处理
- 文件上传时校验格式和大小，不支持的格式即时提示

**数据可视化：**

- 知识图谱支持全屏展示
- 提供图例说明所有视觉编码（颜色、大小、线型）
- 整合报告支持导出为 Markdown 格式

> **注意**：PNG/SVG 图片导出和多角色支持（教师/学生/管理员视角）为未来规划功能，详见文末"未来规划"章节。

### 3.3 可扩展性考虑

**教材扩展：**

- 系统支持动态添加新教材，无需修改核心逻辑
- 新教材上传后自动进入解析->提取->整合流程
- 整合结果自动更新，新教材知识点纳入已有知识图谱
- 当前设计支持 7 本教材，架构可扩展至 20+ 本

**学科扩展：**

- 知识点分类体系（6类）可配置扩展
- Prompt 模板按教材类型管理，新增学科只需添加对应模板
- 知识图谱的关系类型支持自定义扩展

**模型扩展：**

- Embedding 模型通过接口抽象，可替换为其他模型（如 M3E、text2vec-chinese）
- LLM 调用通过统一接口封装，可切换 DeepSeek/Qianwen/GPT 等
- 向量检索可从当前 NumPy 余弦相似度迁移至 FAISS/Milvus/Pinecone 等高性能向量数据库方案

**部署扩展：**

- 当前为单机部署，前端静态资源 + 后端 API 服务
- 可扩展为容器化部署（Docker），支持水平扩展
- 向量索引和文件存储可独立部署为微服务

### 3.4 可靠性需求

- API 调用实现指数退避重试，最大重试 3 次
- 处理进度持久化到本地存储，支持断点续传
- 知识库定期备份，支持回滚到历史版本
- 日志系统记录所有关键操作，便于问题排查

### 3.5 安全性需求

- API Key 通过环境变量注入，不硬编码在代码中
- 用户输入进行 XSS 和注入防护
- 文件上传限制格式（仅 PDF）和大小（单文件 <= 200MB）
- 敏感配置信息不暴露在前端代码中

## 附录：系统约束与假设

**约束条件：**

1. 输入教材为 PDF 格式，且为文字版（非纯扫描图片）
2. 单本教材不超过 1000 页，单文件不超过 200MB
3. API 调用受服务商并发和频率限制
4. 前端运行环境要求现代浏览器（Chrome/Firefox/Edge 最新版）

**基本假设：**

1. 教材内容为标准中文医学教材，术语使用规范
2. 7本教材覆盖基础医学到临床医学的主流学科
3. 用户具备基本的计算机操作能力
4. 网络环境稳定，API 服务可用性 >= 99%

## 未来规划

> **以下功能在本文档的需求分析中有所描述，但在当前版本中尚未实现。** 它们记录了系统的演进方向，供后续开发参考。

### F1. OCR 预处理支持

- **当前状态**：不支持扫描版 PDF（纯图片无文字层）
- **规划方案**：集成 OCR 引擎（如 PaddleOCR），对扫描版 PDF 进行文字识别后再进入正常解析流程
- **优先级**：中

### F2. 撤销与重做 (Undo/Redo)

- **当前状态**：教师对整合决策的修改为直接生效，不支持撤销
- **规划方案**：在教师审核界面中实现操作历史栈，支持单步撤销和重做
- **优先级**：低

### F3. 多角色权限管理

- **当前状态**：系统为单用户模式，无角色区分
- **规划方案**：实现教师/学生/管理员三种角色，各自拥有不同的功能权限（教师：整合审核；学生：RAG问答与图谱浏览；管理员：系统配置与用户管理）
- **优先级**：低

### F4. 图谱导出为 PNG/SVG 图片

- **当前状态**：知识图谱仅支持浏览器内交互查看，不支持图片导出
- **规划方案**：利用 D3.js 的 Canvas/SVG 导出能力，支持将当前视图导出为 PNG 或 SVG 格式
- **优先级**：低

### F5. 页眉页脚自动过滤

- **当前状态**：PDF 解析时未对页眉页脚进行自动过滤，可能混入正文
- **规划方案**：基于位置判断（页面顶部/底部 15% 区域）、长度过滤和相邻页比对等策略，自动识别并去除页眉页脚文本
- **优先级**：中
